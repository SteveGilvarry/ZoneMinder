Project Roadmap

ZoneMinder Modernization Roadmap

A typed, API-centric future for ZoneMinder — one documented API, modern clients, and an optional next-generation engine. Last updated June 2026.

1. Vision

Move ZoneMinder onto a typed, API-centric architecture: one service in front of everything, modern clients that all speak a single documented API, and — eventually and optionally — a new engine for capture and analysis.

ZoneMinder has earned its longevity, but its foundations show their age. The web UI and REST API sit on PHP + CakePHP 2.x, a framework that has reached end of life. Daemon control and much of the glue is Perl. The capture/analysis/streaming core is C++, tightly coupled to its current data and process model. Today's clients — mobile apps, Home Assistant, single-page dashboards — are awkward to build against because there is no single, well-defined surface to target.

This roadmap introduces a single service (zm-api) as the control, data, and streaming plane; modern web and native clients that all speak one documented API; and — eventually and optionally — a new engine (zm-next) for capture and analysis. The aim is not novelty for its own sake. It is to retire dead-end platforms and give every client a single, well-tested surface instead of three languages each reaching into the database.

2. Target architecture

Today there are many entry points and three languages touching the database directly. The target collapses that to one API in front of everything — one auth model, one surface, one place to add features. Clients stop needing to know how capture works.

4. The stages

Stage 1 — current focus

zm-api reaches a first release and takes over the API, daemon, and streaming

zm-api replaces the CakePHP 2.x API, takes over the daemon layer (zmdc/zmpkg/zmu, via a manager and a compatibility socket), and serves live and recorded streaming (HLS/WebRTC/VOD, replacing zms/nph-zms). zmNinjaNg cuts over to the new API.

Rollout: ship passive first — the API runs alongside stock ZoneMinder, REST-only — then flip to active takeover of daemon control once proven on real installs. Passive-first is the key de-risker for replacing daemon control on production systems.

Stage 2 — next

zm-dashboard replaces the PHP frontend

The React zm-dashboard becomes the web UI, talking only to zm-api — no PHP component touches the database any more. Its dual-skin design (a modern view plus a classic-compatible view) smooths the transition. Once it reaches parity, CakePHP and the PHP classic skin are retired. A single sign-on bridge during the coexistence window keeps users from juggling two logins.

Stage 3 — optional / long horizon

zm-next replaces the C++ capture/analysis core

The plugin-pipeline worker zm-next replaces zmc/zma, supervised by zm-api. This is deliberately the optional endgame. The engine is largely built — capture, decode, motion, detection, tracking, recording, and streaming all run as hot-swappable plugins, validated end-to-end on file inputs. The remaining work is the hard part of any new core: validation on real cameras at scale, recording hardening, broader GPU support, and per-camera cutover with legacy fallback. It does not block Stages 1–2 — the modernization stands on its own even if the existing C++ core is kept indefinitely.

Parallel — client track

Mobile (Apple/Android) + Home Assistant integration

The native mobile apps and a new Home Assistant integration are independent clients that all consume zm-api. They move in parallel, once there is a stable API surface to build against. zmNinjaNg simply repoints at the new API as part of Stage 1.

5. Dependency & sequencing

Nearly everything turns on one event: zm-api's first solid release. zmNinjaNg only needs to repoint; the dashboard, mobile apps, and Home Assistant integration build on a stable API surface. zm-next runs on its own long-horizon, optional track. This is a sequence of milestones, not a dated schedule.

6. Design principles

  • One API, one auth model. Every client — web, mobile, Home Assistant, third-party — speaks the same documented surface. No client reaches into the database.
  • Coexist, then cut over. New components run alongside the existing system first and take over only once proven, with a clear path back. No flag-day migrations.
  • Single sign-on across the transition. While the new and old UIs coexist, users get one login, not two.
  • The C++ core replacement is optional. The modernization delivers its value through Stages 1–2 alone; zm-next is a long-horizon bonus, not a prerequisite.
  • Open source, sustainably. The new components are AGPL-licensed (zm-api and zm-next are dual-licensed AGPL/commercial), carrying forward ZoneMinder's copyleft lineage.

7. How to follow along & contribute

The near-term focus is getting zm-api to its first release — that single milestone unblocks the dashboard, the mobile apps, and the Home Assistant integration. The most useful contributions right now:

  • Try zm-api in passive mode alongside an existing install and report what does and doesn't match the current behaviour.
  • Test zmNinjaNg and the dashboard against zm-api, especially live and recorded streaming on real devices.
  • Help with the Home Assistant integration, which is open for community ownership.
  • Exercise zm-next on real cameras and hardware if you're interested in the next-generation engine.

Open questions still being worked through with the community include the eventual home of these repositories under the ZoneMinder organisation, per-stage release targets, and the shape of the auth bridge during the UI transition. Feedback is welcome through the usual ZoneMinder channels.

This roadmap describes intended direction, not a commitment to dates or specific versions. Status reflects the projects as of mid-2026 and will evolve.
