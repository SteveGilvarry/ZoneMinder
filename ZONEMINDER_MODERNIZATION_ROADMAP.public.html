01 — Vision

One surface in front of everything

Move ZoneMinder onto a typed, API-centric architecture: one service in front of everything, modern clients that all speak a single documented API, and — eventually and optionally — a new engine for capture and analysis.

ZoneMinder has earned its longevity, but its foundations show their age. The classic web interface is plain PHP — hand-written pages, AJAX endpoints, and a front controller. The REST API is a separate CakePHP 2.x application, built on a framework that has reached end of life. Daemon control and glue is Perl. The capture, analysis, and streaming core is C++. Four distinct layers, three languages, and several of them reaching straight into the database.

This roadmap introduces a single service (zm-api) as the control, data, and streaming plane; modern web and native clients that all speak one documented API; and — eventually and optionally — a new engine (zm-next) for capture and analysis. The aim is not novelty for its own sake. It is to retire dead-end platforms, give every client a single well-tested surface, and remove the conditions that produced years of security bugs.

03 — Why it matters

A decade of CVEs, almost all in one layer

The case for this architecture is not only ergonomics — it is security. ZoneMinder has more than 50 published CVEs (2013–2024), and they are not spread evenly. They cluster overwhelmingly in the plain-PHP classic web UI; the CakePHP API is a small secondary surface; the C++ daemons and Perl tooling barely appear.

CVE-2019-6991 (zmu)

Honest caveat. New code can carry new bugs, and a typed API is not immune — the CakePHP layer still had a JWT-rooted SQL injection (CVE-2023-26032). The pieces already exist

Most of this future is already in motion, at different stages of maturity.

A next-generation per-monitor worker: a JSON-wired plugin pipeline (capture → decode → motion → detect → track → record / stream), each stage on its own thread, served to zm-api over one socket. ~30 plugins with an embedded AI tier and an optional GPU zero-copy path. Optional and long-horizon; currently in validation and hardening.

Home Assistantplanned
integration

A first-class HA integration that consumes zm-api, alongside the other clients — open for community ownership.

Where things stand: zmNinjaNg has shipped. zm-api is the keystone, in active hardening toward its first release. The dashboard and native apps are ready to move as soon as the API stabilises. zm-next has grown into a substantial engine with a working capture-to-record pipeline and AI tier; what remains is validation and per-camera cutover. The plan turns on one event: zm-api's first solid release.

05 — The stages

How it rolls out

Stage 1 — current focus

zm-api reaches a first release and takes over the API, daemon, and streaming

zm-api replaces the CakePHP 2.x API, takes over the daemon layer (zmdc/zmpkg/zmu), and serves live and recorded streaming (HLS/WebRTC/VOD). zmNinjaNg cuts over to the new API. Rollout is passive-first — the API runs alongside stock ZoneMinder, REST-only — then flips to active takeover of daemon control once proven on real installs.

Stage 2 — next

zm-dashboard replaces the PHP frontend

The React dashboard becomes the web UI, talking only to zm-api — no PHP component touches the database any more. Once it reaches parity, the plain-PHP classic skin and the CakePHP API are retired. A single sign-on bridge during the coexistence window keeps users from juggling two logins.

Stage 3 — optional / long horizon

zm-next replaces the C++ capture/analysis core

The plugin-pipeline worker replaces zmc/zma, supervised by zm-api. Deliberately the optional endgame: the engine is largely built and validated on file inputs; what remains is validation on real cameras at scale, recording hardening, broader GPU support, and per-camera cutover with legacy fallback. It does not block Stages 1–2.

Parallel — client track

Mobile + Home Assistant integration

Native mobile apps and a new Home Assistant integration are independent clients that consume zm-api, moving in parallel once there is a stable API surface to build against.

07 — Principles

What guides the work

  • One API, one auth model. Every client speaks the same documented surface. No client reaches into the database.
  • Coexist, then cut over. New components run alongside the existing system first and take over only once proven, with a clear path back. No flag-day migrations.
  • Security by structure. Eliminate whole vulnerability classes — typed queries, auto-escaped templating, one auth model, memory safety — rather than patching endpoints one at a time.
  • The C++ core replacement is optional. The modernization delivers its value through Stages 1–2 alone; zm-next is a long-horizon bonus, not a prerequisite.
  • Open source, sustainably. The new components are AGPL-licensed (zm-api and zm-next dual-licensed AGPL/commercial), carrying forward ZoneMinder's copyleft lineage.
08 — Contribute

How to follow along & help

The near-term focus is getting zm-api to its first release — that single milestone unblocks the dashboard, the mobile apps, and the Home Assistant integration. The most useful contributions right now:

  • Run zm-api in passive mode alongside an existing install and report what does and doesn't match current behaviour.
  • Test zmNinjaNg and the dashboard against zm-api, especially live and recorded streaming on real devices.
  • Help with the Home Assistant integration, which is open for community ownership.
  • Exercise zm-next on real cameras and hardware if the next-generation engine interests you.

Open questions still being worked through with the community: the eventual home of these repositories under the ZoneMinder organisation, per-stage release targets, and the shape of the auth bridge during the UI transition. Feedback is welcome through the usual ZoneMinder channels.
